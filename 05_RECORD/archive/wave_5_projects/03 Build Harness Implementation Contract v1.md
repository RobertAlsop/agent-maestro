# Agent Maestro — Build Harness Implementation Contract v1

## Purpose

Define the **frozen implementation contract** for **Build Harness v1**.

This contract specifies:

- command surface
    
- command behavior
    
- artifact schemas
    
- identity formats
    
- failure rules
    
- Git discipline
    
- protocol boundaries
    
- verification scope
    

This is the authoritative design artifact for implementing the initial local build harness.

This contract exists to ensure the harness is:

- deterministic
    
- auditable
    
- minimal
    
- safe to operate
    
- cheap in frontier-model token usage
    

---

# Authority Rule

This contract is the **single source of truth** for Build Harness v1.

Implementation must:

- follow this contract exactly
    
- introduce no additional commands
    
- introduce no hidden side effects
    
- not expand scope beyond v1 unless explicitly revised
    

If implementation ambiguity is encountered, the harness must **halt and surface the issue** rather than silently guessing. This follows the project rule of control over autonomy and no silent drift.

---

# Scope

Build Harness v1 includes exactly four commands:

session_start  
context_capture  
session_verify  
session_close

No additional commands are permitted in v1.

No orchestrator wrapper is permitted in v1.

No daemon, background watcher, scheduler, or autonomous runner is permitted in v1.

---

# Environment Contract

Build Harness v1 must assume the following environment:

- OS: macOS
    
- Shell: zsh
    
- Multi-line execution must be safe under `bash -lc`
    
- No network calls by default
    
- Stdlib-only unless explicitly approved
    
- Git available locally
    
- `gh` may or may not be installed
    
- repository is a local Git repository
    

This is consistent with the existing terminal execution rules and build-mode environment discipline.

---

# Implementation Language Contract

Build Harness v1 should be implemented as:

- shell entrypoints
    
- backed by Python stdlib where structured JSON generation or hashing is easier
    

Preferred pattern:

- thin shell launcher
    
- deterministic Python core for structured work
    

However, v1 may also be implemented entirely in Python with shell wrappers.

No non-stdlib Python dependencies are allowed.

---

# Root Location Contract

Build Harness v1 must live under one canonical root.

Approved root options:

scripts/build/

or

tools/build/

One must be chosen and used consistently.

This contract does not force which of the two is chosen, but implementation must pick one and use it everywhere.

No duplicate copies are allowed.

# Identity Contract

## session_id

Format:

YYYYMMDD-HHMMSS

Example:

20260314-181530

Generation rule:

- generated from local system time at session start
    
- second precision
    
- must be zero-padded
    
- must use 24-hour format
    

Collision rule:

- if directory already exists for generated `session_id`, append suffix:
    

YYYYMMDD-HHMMSS-a  
YYYYMMDD-HHMMSS-b

and continue until unique

## run_id

Format:

run-<session_id>

Example:

run-20260314-181530

v1 does not require a separate hash-derived run identity.

## correlation_id

Not required in v1.

May appear later in v2+.

---

# Session Directory Contract

Every session must create one session directory:

06_EXECUTION/logs/sessions/<session_id>/

This directory is mandatory.

All session artifacts must live here.

No session artifact may be written outside this directory except:

- normal repo file modifications produced by the actual build
    
- Git objects written by Git itself
    

---

# Artifact Set Contract

Each session must produce the following files:

06_EXECUTION/logs/sessions/<session_id>/  
  session_manifest.json  
  repo_snapshot.json  
  verification_report.json  
  session_log.md

Optional in v1:

  output_summary.md  
  proposal.md

These may still be generated by the model, but the harness does not require them to exist for command success.

---

# Artifact Schema Contract

## 1. `session_manifest.json`

Required schema:

{  
  "schema_version": "1",  
  "session_id": "20260314-181530",  
  "run_id": "run-20260314-181530",  
  "status": "started",  
  "started_at": "2026-03-14T18:15:30+03:00",  
  "ended_at": null,  
  "branch": "main",  
  "start_commit": "abc123",  
  "end_commit": null,  
  "pre_build_checkpoint_created": false,  
  "verification_status": "not_run",  
  "tag": null,  
  "github_release": null  
}

Allowed `status` values:

started  
verified  
closed  
failed

Allowed `verification_status` values:

not_run  
passed  
failed

## 2. `repo_snapshot.json`

Required schema:

{  
  "schema_version": "1",  
  "session_id": "20260314-181530",  
  "run_id": "run-20260314-181530",  
  "captured_at": "2026-03-14T18:16:00+03:00",  
  "repo_root": "/absolute/path/to/repo",  
  "branch": "main",  
  "head_commit": "abc123",  
  "working_tree_status": "clean",  
  "tracked_file_count": 42,  
  "tree_hash": "sha256-...",  
  "files": [  
    {  
      "path": "README.md",  
      "sha256": "..."  
    }  
  ]  
}

Allowed `working_tree_status` values:

clean  
dirty

Rules:

- `files` must be sorted lexicographically by `path`
    
- only Git-tracked files are included
    
- paths must be repo-relative
    
- `.git/` contents must never be included
    
- symlinks, if present, are listed by path and hashed from target file contents only if readable; otherwise command must fail
    
- binary files are allowed and hashed bytewise
    
- hashing algorithm is SHA-256
    
- `tree_hash` must be computed deterministically from the ordered list of `path + sha256`
    

## 3. `verification_report.json`

Required schema:

{  
  "schema_version": "1",  
  "session_id": "20260314-181530",  
  "run_id": "run-20260314-181530",  
  "verified_at": "2026-03-14T18:45:00+03:00",  
  "status": "passed",  
  "checks": [  
    {  
      "name": "session_manifest_exists",  
      "status": "passed",  
      "details": ""  
    },  
    {  
      "name": "repo_snapshot_exists",  
      "status": "passed",  
      "details": ""  
    }  
  ]  
}

Allowed per-check `status` values:

passed  
failed  
skipped

Allowed top-level `status` values:

passed  
failed

## 4. `session_log.md`

Human-readable markdown summary.

Minimum required sections:

# Session Log  
  
## Session Identity  
## Start State  
## Verification Result  
## Closeout

The harness may append to this file across commands.

If it does not exist, commands that write to it must create it.

# Command Contract

## 1. `session_start`

### Purpose

Initialise a deterministic build session.

### Required behavior

Must:

- confirm current directory is inside a Git repository
    
- generate `session_id`
    
- derive `run_id`
    
- determine repo root
    
- determine current branch
    
- determine current HEAD commit
    
- create session directory
    
- create `session_manifest.json`
    
- create or initialise `session_log.md`
    

Must inspect working tree state using Git.

If working tree is dirty:

- by default, create a pre-build checkpoint commit
    

If working tree is clean:

- do not create pre-build checkpoint commit
    

### Allowed flags

--no-commit

Meaning:

- if repo is dirty, do not auto-commit
    
- instead write manifest with dirty start state and exit non-zero
    

--allow-dirty

Meaning:

- if repo is dirty, allow session to start without checkpoint commit
    
- mark `pre_build_checkpoint_created: false`
    
- record dirty start condition in session log
    

`--no-commit` and `--allow-dirty` must not be used together.

### Default mode

Default behavior is:

- if dirty → `git add -A && git commit -m "Pre-build checkpoint — Session <session_id>"`
    
- if clean → proceed without new commit
    

### Success output

Must print a short machine-usable summary to stdout:

SESSION_ID=<session_id>  
RUN_ID=<run_id>  
SESSION_DIR=<absolute_path>  
START_COMMIT=<commit>  
BRANCH=<branch>

### Exit codes

0 success  
1 general failure  
2 not a git repo  
3 dirty repo and start blocked by flags/policy  
4 session directory creation failure  
5 manifest write failure  
6 git checkpoint failure

---

## 2. `context_capture`

### Purpose

Produce deterministic repo truth artifact.

### Required behavior

Must:

- locate current repo root
    
- locate current active session via explicit flag
    
- write `repo_snapshot.json`
    
- enumerate Git-tracked files only
    
- hash every tracked file using SHA-256
    
- compute deterministic `tree_hash`
    

### Required flag

--session-id <session_id>

No implicit session lookup is allowed.

### File enumeration contract

Must use Git-tracked files only.

Must exclude:

- `.git/`
    
- untracked files
    
- ignored files not tracked by Git
    

### Ordering contract

All file paths must be sorted lexicographically before hashing aggregation and JSON emission.

### Success output

Must print:

SESSION_ID=<session_id>  
SNAPSHOT_FILE=<absolute_path>  
TRACKED_FILE_COUNT=<n>  
TREE_HASH=<sha256>

### Exit codes

0 success  
1 general failure  
2 not a git repo  
3 session not found  
4 file hashing failure  
5 snapshot write failure

---

## 3. `session_verify`

### Purpose

Run deterministic post-build checks.

### Required flag

--session-id <session_id>

### Required checks in v1

The following checks are mandatory:

1. `session_manifest_exists`
    
2. `repo_snapshot_exists`
    
3. `session_log_exists`
    
4. `session_directory_exists`
    
5. `manifest_schema_minimum_fields_present`
    
6. `snapshot_schema_minimum_fields_present`
    
7. `working_tree_readable`
    
8. `git_repo_accessible`
    

### Optional checks in v1

Allowed but not required:

- required project maps exist
    
- registry file exists
    
- no forbidden files created
    
- markdown structure checks
    

These must only be included if implemented deterministically and documented.

### Verification rule

If any mandatory check fails, top-level verification status must be `failed`.

### Success output

Must print:

SESSION_ID=<session_id>  
VERIFICATION_FILE=<absolute_path>  
VERIFICATION_STATUS=passed|failed

### Exit codes

0 verification passed  
1 general failure  
2 session not found  
3 verification failed  
4 report write failure

---

## 4. `session_close`

### Purpose

Close the session with disciplined Git lineage.

### Required flag

--session-id <session_id>

### Optional flags

--tag <tag_name>  
--github-release

Rule:

- `--github-release` may only be used if `--tag` is also provided
    

### Required behavior

Must:

- verify session exists
    
- run no implicit verification
    
- inspect current HEAD commit
    
- `git add -A`
    
- create build commit with exact message format:
    

Agent Maestro build — Session <session_id>

- update `session_manifest.json` with:
    
    - `status: "closed"`
        
    - `ended_at`
        
    - `end_commit`
        
    - tag info if present
        
    - github release status if attempted
        

### Tagging contract

If `--tag` is provided:

- run `git tag <tag_name>`
    
- fail if tag already exists
    
- do not overwrite tags
    

### GitHub release contract

If `--github-release` is provided:

- require `gh` executable to be available
    
- create release for supplied tag
    
- if `gh` unavailable, fail non-zero
    
- no silent skip
    

### Success output

Must print:

SESSION_ID=<session_id>  
END_COMMIT=<commit>  
TAG=<tag_or_none>  
GITHUB_RELEASE=created|not_requested

### Exit codes

0 success  
1 general failure  
2 session not found  
3 git commit failure  
4 tag creation failure  
5 github release failure  
6 manifest update failure

# Git Discipline Contract

Build Harness v1 must obey the following Git rules:

1. No force push behavior
    
2. No push behavior at all in v1
    
3. No rebase behavior in v1
    
4. No branch switching in v1
    
5. No branch creation in v1
    
6. No deletion of tags in v1
    
7. No overwrite of existing tags in v1
    

Important:

- `session_close` creates local commit only
    
- pushing remains manual in v1
    
- if GitHub release is requested, it is allowed only through `gh release create`
    

This keeps v1 safer and smaller.

---

# Failure Policy Contract

All commands must follow these rules:

1. Fail closed
    
2. No silent retries
    
3. No partial success reported as success
    
4. If JSON write fails, command exits non-zero
    
5. If Git command fails, command exits non-zero
    
6. If command fails after partially writing an artifact, it must either:
    
    - overwrite with valid final output before exit, or
        
    - leave partial file absent, not malformed JSON
        

Malformed JSON artifacts are forbidden.

---

# Protocol Boundary Contract

The harness is responsible for:

- session mechanics
    
- repo state capture
    
- verification mechanics
    
- Git closeout mechanics
    

The model is responsible for:

- brief creation
    
- proposal / build contract creation
    
- design reasoning
    
- controlled build reasoning
    
- interpretation of results
    
- session narrative
    

The harness must not:

- generate proposals
    
- decide build scope
    
- redesign architecture
    
- inspect meaning of content beyond deterministic checks
    

The model must not bypass required harness stages once the protocol says a harness stage is mandatory.

This preserves the no-bypass discipline.

---

# Verification Boundary Contract

Build Harness v1 verification is intentionally narrow.

It verifies:

- presence
    
- accessibility
    
- minimum schema integrity
    
- basic repository readability
    

It does not verify:

- architectural correctness
    
- content quality
    
- semantic consistency
    
- roadmap alignment
    
- prompt quality
    
- business logic quality
    

Those remain outside v1.

---

# Logging Contract

Every command must append a short entry to `session_log.md` containing:

- command name
    
- timestamp
    
- result
    
- short details
    

This creates a human-readable event trail.

Example format:

## 2026-03-14T18:16:00+03:00 — context_capture  
Result: success  
Details: repo_snapshot.json written, tracked_file_count=142

---

# Non-Goals for v1

The following are explicitly out of scope:

- automatic push
    
- automatic branch creation
    
- automatic PR creation
    
- diff analysis
    
- semantic repo summarisation
    
- vector memory integration
    
- policy engine integration
    
- autonomous execution chaining
    
- file-content interpretation beyond hashing and minimal checks
    
- live watcher / daemon mode
    
- release artifact attachment automation
    

These can be considered in later versions only.

---

# Deferred Upgrades

The following are intentionally deferred to v2+:

- `correlation_id`
    
- richer context signals in `repo_snapshot.json`
    
- contract hash linkage
    
- proposal file enforcement
    
- stronger schema validation
    
- optional CSV index updating
    
- richer verification checks
    
- release manifest artifact
    
- post-close push support
    
- deterministic context-pack builder beyond raw snapshot
    

This keeps v1 aligned with the principle of starting simple and hardening over time.

---

# Implementation Success Criteria

Build Harness v1 is successful when:

1. A session can be started deterministically
    
2. Repo state can be captured into a stable artifact
    
3. Verification can mechanically confirm minimum session integrity
    
4. A build session can be closed with clean Git lineage
    
5. All artifacts are machine-readable and human-auditable
    
6. The harness reduces model token waste by removing session-administration work
    

---

# Final Statement

Build Harness v1 is not a build intelligence layer.

It is a **deterministic local control harness** around the build session lifecycle.

Its purpose is to reduce frontier-model overhead while increasing:

- repeatability
    
- auditability
    
- state traceability
    
- operational discipline
    

It exists to support the larger governed execution model you are building, not replace it.
