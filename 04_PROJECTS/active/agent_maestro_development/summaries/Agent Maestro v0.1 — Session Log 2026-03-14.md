# Agent Maestro v0.1 — Session Log (2026-03-14)

## Session Summary

Full audit of the Agent Maestro v0.0.1 vault, identification of gaps, design of eight improvements, and resolution of all open design questions. Session concluded with approval to build.

## Key Decisions Made

### 1. Reference Implementation: Vault Curator

**Decision:** The reference constellation is "Vault Curator" — a self-maintaining system that governs the Agent Maestro vault itself.

**Rationale:** Exercises the most framework features (authority packs, compilation, escalation, all three agent layers, structured data CSVs) while solving a real operational need (link integrity, schema validation, orphan detection). Not a toy example.

### 2. Three-Layer vs Two-Layer Memory Model

**Decision:** The architectural principle remains "Three-Layer Memory Model." The current operational state is two active layers (Knowledge Graph + Structured Data), with the Vector Memory Layer architecturally reserved.

**Rationale:** Principles describe design intent, not implementation status. Renaming to "two-layer" would require changing the principle again when a vector database connects. Documentation will clearly distinguish between the architectural target and current implementation.

### 3. Knowledge Graph Implementation

**Decision:** Hybrid approach — Obsidian-native notes as the graph (nodes = notes, edges = wiki-links and frontmatter relationships), with a script-generated CSV index for programmatic access.

**Rationale:** The vault already IS a knowledge graph implicitly. Making it explicit requires populating the `relationships` frontmatter blocks and generating a CSV export. Zero new dependencies. Compatible with Google Sheets. Obsidian graph view works natively.

### 4. Structured Data Layer: CSV with Google Sheets Compatibility

**Decision:** CSV files with defined conventions (UTF-8, comma delimiter, ISO dates, atomic values, underscore filenames).

**Rationale:** Google Sheets imports/exports CSV natively. No conversion layer needed. Eight specific CSV files defined, each serving a distinct operational concern.

### 5. CLAUDE.md Governance Files

**Decision:** Place CLAUDE.md files in the vault root, all seven top-level folders, and four critical subdirectories (14 total).

**Rationale:** This is Agent Maestro's governance philosophy applied to itself — governance before autonomy. Before an agent works in a folder, a contract defines the rules. CLAUDE.md files are lightweight governance contracts complementing the full authority packs.

### 6. Agent Maestro Layer Model + MWP

**Origin:** Proposed by the vault owner during the session.

**Decision:** Introduce a four-layer runtime model (Authority → Agent Runtime → Workflow → Workspace) with the Model Workspace Protocol (MWP) defining standardised workspace structure for agent execution.

**Rationale:** Fills the gap between governance intent and execution mechanics. Makes every agent execution self-contained and auditable. The Vault Curator reference implementation will use MWP workspaces.

### 7. Compiler Dependencies

**Decision:** Use `pyyaml` and `jsonschema`.

**Rationale:** The compiler validates governance — it should use proven tools. Reimplementing parsers introduces bugs into the governance layer, violating Principle 3.

### 8. Knowledge Graph CSV Generation

**Decision:** Generated by script on demand, not committed as a static file.

**Rationale:** A committed CSV drifts from frontmatter when notes are edited. On-demand generation guarantees accuracy.

### 9. Vault Curator Specialists

**Decision:** Implemented as runnable Python scripts with authority packs.

**Rationale:** Without scripts, the reference implementation is documentation describing documentation. With scripts, you can actually run the system and verify the framework works.

### 10. Build Model

**Decision:** Use Opus for the build.

**Rationale:** Cross-file consistency across 50+ interconnected deliverables (frontmatter, wiki-links, relationship fields, Python scripts, JSON schemas) requires Opus's stronger reasoning. The proposal serves as the build contract — no additional planning document needed.

### 11. CLAUDE.md Frontmatter Exception

**Decision:** CLAUDE.md files do not use YAML frontmatter.

**Rationale:** They are infrastructure files, not vault notes. Adding frontmatter would make them appear in the knowledge graph and schema validator as knowledge entities, which they are not. The root CLAUDE.md now explicitly documents this exception.

### 12. Quickstart Guide Clarification

**Decision:** Changed "Build Your First Constellation" heading to "Explore the Vault Curator Constellation."

**Rationale:** The Vault Curator is already built. The Quickstart walks readers through the existing system, not a build-from-zero tutorial. The original heading was misleading.
